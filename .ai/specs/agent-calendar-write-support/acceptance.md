# 功能验收标准

- 当用户对 agent 说出完整课程信息并要求“写进日历/课表”时，agent 会先展示确认文案；用户确认后，课程成功写入个人课表。
- 当用户对 agent 说出考试信息并要求“记进日历”时，agent 会先展示确认文案；用户确认后，考试成功写入一次性事件表。
- 当用户信息不完整时，agent 会继续追问缺失字段，而不是直接失败或直接写入。
- 同一条考试事件重复创建时，系统不会生成明显重复的多条记录。
- 用户回到个人课表区域时，仍能正常看到原有课程课表；并能看到新增的 upcoming 考试/事件区块。
- 原有 `read_user_schedule` 课表查询仍然能正常回答今天/明天/下一节课。

# 边界条件

- 用户只说“帮我记个考试”但没给日期：必须继续追问，不能写入。
- 用户只说课程代码但没有时间信息：必须继续追问，不能写入。
- 用户提供的是模糊自然语言日期，如“下周三”：首期实现前需明确是否支持；若不支持，agent 必须要求用户改成明确日期。
- 用户取消确认或未确认：不得发生落库写入。
- 用户未登录：不得写入，需提示登录。

# 回归检查点

- OCR 导课保存到 `user_schedule_entries` 的链路不受影响。
- 手动新增课程课表仍然可用。
- 课表编辑、删除仍然可用。
- 课程收藏自动补齐逻辑与本任务无耦合，不应被影响。
- widget 同步课表仍然可用，不因新增考试事件报错。

# 失败判定条件

- agent 在未确认前就写入课表或考试事件。
- 考试事件被错误写入 `user_schedule_entries`。
- 现有课程课表加载失败、OCR 导课失败或课表编辑失败。
- upcoming 事件区块读取导致 `MyScheduleCard` 空白、崩溃或无限加载。
- 重复执行同一考试写入命令后产生多条重复事件。

# 人工验证步骤

1. 登录测试账号，打开 agent。
2. 输入“帮我把 COMP3015 每周二 09:00-10:50 记进日历，地点 AAB201”。
3. 验证 agent 先返回确认文案，且文案包含课程名/课程代码、星期、时间、地点。
4. 回复“确认”。
5. 打开个人课表区域，验证该课程出现在对应星期的课表列表中。
6. 输入“我 2026-05-12 14:00 有 COMP3015 final，地点 WLB204，帮我顺手记进日历”。
7. 验证 agent 先返回确认文案，且文案包含日期、时间、地点、考试标题。
8. 回复“确认”。
9. 打开个人课表区域，验证“即将到来的日程 / 考试”区块出现该事件。
10. 再次发送同样的考试记忆请求并确认，验证不会出现明显重复记录。
11. 测试缺失字段场景，如“帮我记个考试”，验证 agent 会追问日期/时间而不是直接写入。
12. 重新测试“我今天有什么课”，验证原课表读取回答仍然正常。
