# 背景

当前项目里的 agent 只能读取用户课表，不能替用户新增课表/日历内容。现有个人时间相关能力主要集中在两条链路：

- `services/schedule.ts` + `components/profile/MyScheduleCard.tsx`：管理重复出现的课程课表。
- `services/agent/executor.ts` + `services/agent/tools.ts`：agent 只能通过 `read_user_schedule` 读取课表摘要，不能执行写操作。

同时，现有 `user_schedule_entries` 结构适合“按星期重复出现的课程”，不适合“某年某月某日的一次性考试/事件”。如果直接把考试硬塞进现有课表结构，会把重复课程和一次性事件混在一起，后续查询、展示、去重和编辑都会变复杂。

# 目标

让 agent 支持两类写入能力，并统一走明确确认后再落库的流程：

1. 用户让 agent “帮我把这门课写进日历/课表” 时，agent 能创建课程课表项。
2. 用户在对话里提到“某天有考试/quiz/midterm/final/presentation”并要求顺手记进日历时，agent 能创建一次性考试事件。
3. 写入后的数据可以被现有个人时间体系读取，至少能被 agent 再次查询到，并在个人课表区域有基础可见性。

# 非目标

- 不实现 Apple Calendar / Google Calendar / Outlook 等系统日历同步。
- 不重构 OCR 导课、课程收藏、widget 同步、课程社区等无关链路。
- 不在本次任务里实现复杂提醒系统、推送调度、ICS 导出、跨设备日历订阅。
- 不把所有时间数据统一重构成全新“超级日历模型”。

# 输入条件

- 用户已登录，存在 `userId`。
- 现有 `services/schedule.ts`、`components/profile/MyScheduleCard.tsx`、`services/agent/executor.ts` 可作为复用基础。
- agent 对课程写入和考试写入都必须通过显式确认后再执行，不能自动落库。
- 用户口述的信息可能不完整，agent 需要补齐最少必填字段后才能写入。

# 输出结果

- 形成一套最小改动的实现蓝图：
  - 课程写入继续复用现有课表模型；
  - 考试写入使用新的“一次性日历事件”模型；
  - agent 增加相应写工具与确认流；
  - 个人课表区域增加对一次性事件的基础展示。
- 输出可直接交给实现代理落地的 5 份 spec 文件。

# 关键假设

- “写入日历”在本次任务中优先解释为“写入 App 内个人时间管理数据”，而不是写入手机系统日历。
- 课程仍然以“按星期重复”的课表项存储，考试仍然以“指定日期的一次性事件”存储。
- `MyScheduleCard` 是当前最合理的首个展示入口，因此不新建独立页面。
- 如果用户没有提供完整时间信息，agent 可以继续追问，但在缺少必要字段时不能写入。

# 约束条件

- 必须保持现有 `user_schedule_entries` 兼容，不破坏 OCR 导课、手动加课、编辑课表、widget 同步。
- 必须把“重复课程”和“一次性考试事件”分开建模，避免把一次性日期塞进按周重复结构。
- agent 写入必须沿用当前 executor 的“待确认写操作”风格，不能新增绕过确认的直写通道。
- 变更范围应限制在 agent、时间数据服务、个人课表展示和必要测试；不触碰无关模块。
