# 第 1 步：先落地 App 内部日历写入

## 目标

先让 agent 稳定支持两类写入：

- 重复课程课表写入
- 一次性考试 / quiz / presentation 事件写入

并保证这两类数据都能在 App 内被再次读取、展示和确认。

## 本阶段范围

- 沿用 `user_schedule_entries` 存课程课表
- 新增 `user_calendar_events` 存一次性事件
- agent 支持补问、确认、写入
- 个人课表区域支持展示 upcoming 一次性事件

## 本阶段成功标准

- 用户可以通过 agent 记课程
- 用户可以通过 agent 记考试
- agent 再次查询时能读到这些内容
- UI 能看到写入结果
- 不影响现有 OCR 导课和手动课表编辑

## 为什么先做这一步

- 不依赖系统权限
- 不受 iOS / Android 平台差异影响
- 更容易控制去重、更新、删除逻辑
- 可以先验证“用户到底会怎么说、怎么用”这件事

# 第 2 步：抽象统一的时间事件接口

## 目标

在内部数据已经跑顺之后，把“重复课程”和“一次性事件”整理成一层统一的时间事件抽象，为后续系统日历同步做准备。

## 本阶段范围

- 在服务层增加统一的时间事件读取模型，例如：
  - `calendar item`
  - `schedule occurrence`
  - `upcoming agenda item`
- 统一 agent 侧读取与展示语义，减少“课程表”和“考试事件”两套口径分裂
- 明确每类事件的 source、重复规则、编辑能力、去重规则

## 建议输出

- 一个统一的事件视图模型，而不是立刻重构底层表
- 一组稳定字段，例如：
  - `id`
  - `kind`
  - `title`
  - `date`
  - `startTime`
  - `endTime`
  - `location`
  - `source`
  - `isRecurring`

## 本阶段成功标准

- App 内能用同一套视图逻辑读取“课程 + 考试”
- agent 可以更自然地回答“我接下来有什么安排”
- 为第 3 步系统日历同步提供稳定适配层

## 为什么第 2 步先做抽象、不急着接系统日历

- 如果没有统一接口，后续系统日历同步会直接绑定到零散内部表结构
- 一旦将来内部结构变化，系统日历同步层会很难维护
- 先抽象再同步，可以把外部集成风险压到最低
